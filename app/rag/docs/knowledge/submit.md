# Submit 提交模块

## 模块职责

`submit` 模块负责的是“提交记录的生成、提交源码的对象存储、提交结果的查询展示，以及提交完成后与判题链路、榜单链路的衔接”。它本身不承担判题执行，也不直接改写榜单数据，而是把一次提交拆成几段明确的职责：先完成请求校验和权限校验，再把提交主记录落到 PostgreSQL，随后把源码写入 MinIO，最后通过 RabbitMQ 把提交编号投递给后续判题消费者。这个模块的查询能力也围绕同一份提交主记录展开，提交详情会从数据库读取测评摘要、从 MinIO 读取源码、从明细表读取测试点结果，而提交列表则只返回可公开展示的摘要字段，不返回源码和测试点细节。

从业务边界上看，`submission` 记录的是一次提交行为的事实，而不是“最终成绩单”。`judge` 是当前 HTTP 层直接暴露的提交入口，数据库层已经预留了 `judge`、`hack`、`run`、`check` 这类任务类型字段，但当前接口实现的主线仍然是常规测评提交。对于 Agent 提交，系统会把触发提交的登录用户写入 `submitter_id`，再用 `is_agent` 区分来源，这样既能保留归属关系，也能在列表和统计中按业务口径过滤掉 Agent 记录。

## 对外接口

`POST /submit/judge` 是提交代码的入口，`problemId`、`language`、`code`、`isAgent` 为必填字段，`contestId` 仅在比赛上下文中使用。接口会先做参数级校验，再做题目存在性和提交权限校验；校验通过后返回一个 `submissionId`，后续页面刷新、轮询提交详情或关联判题结果都依赖这个编号。`GET /submit/{submissionId}` 用来查询单次提交的完整详情，返回题目、语言、状态、耗时、内存、整体信息、源码和测试点明细，但权限只开放给提交者本人、管理员和超级管理员。`GET /submit/contest/{contestId}/user/{userId}` 用来查看某个用户在某场比赛中的提交历史，不分页；`GET /submit/user/{userId}` 用来分页查看某个用户的全部提交历史，包含比赛内和比赛外的提交。

列表接口和详情接口的返回边界是分开的。详情接口可以返回源码和测试点摘要，因为它本身已经做了身份校验；列表接口只返回 `submissionId`、`problemName`、`language`、`status`、`time`、`memory`、`submitTime`，不会把源码或测试点明细带出来。`problemName` 不是直接透传数据库字段，而是服务端根据当前题目数据重新拼接后的展示名：比赛内格式固定为 `A. name`，普通用户提交列表格式固定为 `#123. name`。如果题目元数据已经不存在，列表里就返回 `null`，不人为补一个容易误导的占位文本。

## 提交流程

一次 `judge` 提交的执行顺序是固定的。请求进入 `SubmitService.judge()` 之后，先由 `SubmitRequestChecker` 做结构校验，过滤掉空请求、无效题号、无效比赛号、非法语言、空代码和缺失的 `isAgent` 标记。随后系统根据 `problemId` 查询题目记录；如果题目不存在，直接返回 `PROBLEM_NOT_FOUND`。题目存在后，再交给 `ProblemPermissionChecker.checkSubmitPermission(...)` 做提交权限判断，这一步会把题目可见性、比赛上下文、报名状态以及 Agent 边界一起纳入判断，避免只校验“能不能看题”却放过“能不能提交”的情况。

权限通过后，服务会立即构造 `submission` 主记录，并把 `type` 设为 `judge`，把 `submitterId` 设为当前登录用户，把 `isAgent`、`contestId`、`language` 一并写入，同时把初始状态设为 `PENDING`。提交时间由应用侧直接调用 `LocalDateTime.now()` 生成，避免数据库时间和应用时间混用，后续比赛封榜、提交列表排序和榜单重算都按这份提交时间作为统一依据。主记录先入库，再把源码上传到 MinIO 的 `submission/{submissionId}/code` 对象；最后通过 RabbitMQ 发送提交编号，routing key 固定为 `submit`，让后续判题消费者继续处理。这条链路当前是“数据库、对象存储、消息队列”串联的方式，数据库事务只覆盖本地写入，MinIO 上传和 MQ 投递属于外部系统操作，因此它们的失败不会被数据库自动回滚清理掉，文档和排障时需要把这点当成明确的一致性边界来看。

## 数据结构

`submission` 是提交主表，字段包括提交类型、题目 ID、提交者 ID、是否 Agent、提交时间、比赛 ID、语言、状态、最大运行时间、最大内存占用和整体测评信息。模块内真正被查询接口使用的字段主要是 `problemId`、`submitterId`、`contestId`、`isAgent`、`submitTime`、`language`、`status`、`timeUsedMs`、`memoryUsedMb` 和 `info`；其中 `timeUsedMs`、`memoryUsedMb` 和 `info` 会在判题结束后由后续链路回填。`submission_detail` 是测试点明细表，主键是 `submission_id + test_case_id`，记录每个测试点的输入摘要、输出摘要、标准答案摘要、错误信息、状态、运行时间和内存占用。这里的“摘要”是服务端用于展示和排错的短内容，不是整份测试数据文件。

MinIO 中存放的是提交源码本体，路径固定为 `submission/{submissionId}/code`，语言信息不放在对象名里，而是单独保存在 `submission.language`。查询提交详情时，服务会先读数据库，再按这个路径回读源码；如果数据库里有提交记录，但 MinIO 对象读取失败，当前实现会把它当成系统异常处理，而不是静默返回一个空源码，这样可以显式暴露数据库和对象存储之间的不一致。

## 查询与可见性

提交详情查询要求用户先登录，然后再判断是否有权限读取这条提交。权限规则很直接：提交者本人可以看，管理员和超级管理员也可以看，其他人直接返回 `FORBIDDEN`。如果提交关联了比赛，并且查看者不是管理员，那么系统还会根据比赛结束时间决定是否展示测试点明细。比赛未结束时，提交详情里的源码仍然可见，但测试点明细会被隐藏成空列表；比赛结束后，或者这条提交本来就不属于任何比赛时，测试点明细才会完整返回。题目已经被删除或下线时，提交详情仍然保留，只是 `problemName` 会变成空值，保证历史提交还能被正常打开，而不会因为题目元数据缺失导致整条记录不可读。

比赛维度的提交列表是公开查询接口，但它只返回非 Agent 提交。列表服务先按 `contestId` 和 `submitterId` 取出提交记录，再根据访问者身份决定是否隐藏封榜后的提交：如果当前访问者不是提交者本人，也不是管理员，就会在比赛处于封榜区间时过滤掉 `submitTime >= freezeTime` 的记录，边界点按照提交时间严格处理，提交时间正好等于 `freezeTime` 也会被视为封榜后提交并隐藏。然后服务再根据当前比赛的 `contest_problem` 编排和题目名称组装展示名，确保展示出来的 `A. name` 对应的是这场比赛当下的题号，而不是简单地拿题目 ID 拼接。

用户全部提交列表则是分页接口，只有提交者本人、管理员和超级管理员可以查看。这个接口同样只返回非 Agent 提交，查询层会先按 `submitterId` 过滤，再按 `submitTime DESC, id DESC` 排序，并且限制每页最多 100 条。分页返回体沿用标准的分页结构，`records` 只装提交摘要，`total`、`current`、`size` 和 `pages` 直接透传分页结果。这里的展示名按题目 ID 重新拼成 `#123. name`，其含义是“这条提交对应的是哪一道题”，而不是“它在某场比赛里的编号是什么”。

## 与判题和榜单的联动

`submit` 模块生成的提交编号，会继续流向判题链路和榜单链路。`Producer` 负责把提交编号投递到 `tenjudge.judge.exchange`，路由键固定为 `submit`；后续判题完成后，消费者会把结果消息送到 `tenjudge.judge.complete.queue`，由 `Listener` 收到后调用 `BoardService.handleJudgeResult(submissionId)`。也就是说，这个模块只负责把“有一条新提交”这件事可靠地发出去，真正的判题结果回写和榜单刷新不在这里完成。

从数据视角看，列表和详情接口都默认排除 `isAgent = true` 的提交，这样可以避免 Agent 提交混入普通用户历史记录。Agent 的存在更多是为了支持自动化构造测评输入、复现问题或辅助验证，而不是作为普通提交历史的一部分。对于比赛场景，`ProblemPermissionChecker` 还会额外约束 Agent 提交不能绕过私有题和报名限制，这一点是提交安全边界的一部分，不能只看前端传没传 `contestId`。

## 异常与边界

这个模块里常见的业务异常主要有三类：请求参数非法时返回 `SUBMIT_REQUEST_INVALID`，题目不存在时返回 `PROBLEM_NOT_FOUND`，提交记录不存在或访问者无权查看时返回 `SUBMISSION_NOT_FOUND` 或 `FORBIDDEN`。源码上传到 MinIO 失败、读取源码失败、以及消息投递失败则会以系统异常形式抛出，统一交给全局异常处理器处理。这样的划分保持了业务错误和基础设施错误的边界，也便于在日志和接口返回里快速区分“请求本身不合法”与“外部依赖链路出错”。
